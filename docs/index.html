Production Architecture

YouTube Sliding Top-K Rankings

A first-party ranking system for top watched videos over sliding windows. The API does not compute top-k from raw events. It reads materialized ranking snapshots produced by a streaming pipeline.

Sliding Windows5m, 1h, 24h, and 7d windows with event-time bucket state.
Validated ViewsRankings use accepted view events, not raw playback starts.
Precomputed Top-KShard-local candidate lists merge into global snapshots.
Watermarked OutputExact for processed validated events up to the watermark.

Component Navigation

Use this as the root map for the system. Component 3 has its own detailed page now; the other cards jump to the matching root-page section until we split them out.

1 Playback Clients Web, mobile, TV, and embedded clients emit raw watch events. 2 Watch Event Gateway Ingress boundary for authentication, schema checks, and rate limits. 3 Validation / Dedup / Fraud Trust boundary that turns raw playback into ranking signals. 4 Event Stream Kafka or Pub/Sub transport for valid, rejected, suspicious, and late events. 5 Normalizer / Enricher Adds dimensions such as country, category, creator, and language. 6 Sliding Aggregator Maintains event-time bucket counters for rolling windows. 7 Top-K Ranker Computes shard-local candidates and global ranking snapshots. 8 Serving + Storage Snapshot stores, Redis/edge cache, metadata hydration, and the Top-K API.

2. Write Path: Event To Snapshot

Every view must pass validation before it can change a ranking. The ranker publishes snapshots on a cadence, not on every event.

  Client->>Gateway: WatchEvent(videoId, user/session, eventTime, watchMs)
  Gateway->>Validator: Authenticated raw event
  Validator->>Validator: Deduplicate + classify valid view
  Validator->>Log: ValidViewEvent
  Log->>Enricher: Partitioned stream event
  Enricher->>Agg: Ranking dimensions added
  Agg->>Agg: Assign event-time bucket and update rolling score
  Agg->>Ranker: Updated video score
  Ranker->>Merger: Local top 1000 per ranking definition
  Merger->>Store: Durable RankingSnapshot(asOf, watermark)
  Merger->>Cache: Hot snapshot for API reads

4. Sliding Window Top-K Internals

Sliding windows are maintained as event-time buckets. Scores can increase when new events arrive and decrease when buckets expire, so top-k state needs periodic rebuilds or lazy invalidation rather than a simple append-only heap.

7. API Contract

The API response must state the ranking definition, window, bucket size, watermark, and quality. That prevents consumers from confusing raw playbacks, validated views, and public counters.

Principal engineering decisionThe serving API never scans raw events. It reads a materialized ranking snapshot and returns freshness metadata.
GET /v1/top-videos
  ?scope=country:US
  &window=1h
  &metric=valid_views
  &k=100

8. Implementation Boundaries

The Java implementation should use these as module/interface boundaries even if the first version uses local adapters before Kafka, Flink, Redis, and durable stores.

Input

Watch Event Ingestion

Schema validation, auth, duplicate event suppression, low-quality playback filtering, and late-event policy.

State

Sliding Window Aggregation

Event-time bucket assignment, rolling score maps, watermark tracking, and bucket expiration.

Ranking

Top-K Snapshotting

Shard-local candidate sets, global merge, deterministic tie-breaking, snapshot versioning, and cache publication.

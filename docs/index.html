Production Architecture

YouTube Sliding Top-K Rankings

A first-party ranking system for top watched videos over bucket-aligned sliding windows. Raw playback is durably captured before filtering. The API reads materialized snapshots with explicit watermark, bucket size, and rank quality metadata.

Raw Durability FirstRaw events are retained before validation can reject or quarantine them.
Exact BucketedRankings are exact over full event-time buckets up to the watermark.
CorrectionsFraud and policy systems can emit negative deltas and suppression events.
Safe Top-KSalted hot-key partials are combined by video before final ranking.

2. Write Path With Raw Replay

A bad validation deployment should not permanently lose raw activity. Raw events are written first, then compacted into view candidates for Component 3.

Event To Snapshot With Replay

  participant Log as C4 Topics

  participant Rank as C7 Ranker

  Client->>Gateway: raw playback heartbeat or watch segment
  Gateway->>Raw: append immutable raw event
  Raw->>Compact: compact noisy heartbeats into ViewCandidateEvent
  Compact->>Validator: candidate with rawEventRef and traceId
  Validator->>Log: valid / rejected / suspicious / late outcome
  Log->>Agg: accepted events and correction deltas
  Agg->>Rank: bucketed metric updates
  Rank->>Store: snapshot revision with asOf and watermark

4. Bucketed Sliding Window Contract

The public contract is bucket-aligned exactness, not magical per-event exactness for arbitrary times.

Contract

exact_bucketed

For asOf=12:00:00, a 1h window means [11:00:00, 12:00:00) over complete buckets.

Watermark

Event-time completeness

A snapshot includes all accepted events with event_time <= watermark.

Corrections

Revisioned snapshots

Late fraud and policy changes emit correction deltas and produce new snapshot revisions.

5. Sliding Aggregation And Safe Top-K

Hot-key salting is necessary, but salted partials must be combined by video before final ranking. Otherwise top-k is wrong for viral videos.

Salted Partial Combine Before Top-K

  bucket[Bucket counters per rankingDefinitionId + bucketStart + videoId + salt]
  combine[Combine partials by videoId]

  global[Global top-k merge]

  correction --> partial
  partial --> bucket --> combine --> owner --> local --> global --> snapshot

7. Serving Contract And Stale Behavior

The API reads materialized snapshots and states freshness explicitly. It does not scan events.

Serving fallbackCache hit serves immediately. Cache miss reads the snapshot store. A stale snapshot can be served with stale=true until maxStaleness; beyond that the API returns degraded or unavailable.
GET /v1/top-videos?scope=country:US&window=1h&metric=valid_views&k=100

Sliding Window Strategy

Use micro-bucketed sliding windows. This gives bounded error, predictable memory, fast expiry, and a clean path to event-time watermarks and correction snapshots.

Top-K Computation

Scores rise when new events arrive and fall when buckets expire. The ranker keeps score state separate from candidate state and periodically rebuilds candidates.

1

Shard-local state

Each shard owns a subset of videos and tracks rolling scores for registered scopes such as global and country.

2

Candidate top-k

Each shard emits an internal top 1000 for a public top 100 to keep global merges stable.

3

Global merge

The merger combines only shard candidates, applies deterministic tie-breaking, and writes a versioned snapshot.

API Contract

The API explains what the ranking means. The count is exact for accepted events processed up to the watermark, not a claim about every raw playback in the world at this instant.

Principal decision
The serving API does not compute rankings. It resolves a registered ranking definition, reads a snapshot, hydrates metadata, and returns freshness metadata.
GET /v1/top-videos
  ?scope=country:US
  &window=1h
  &metric=valid_views
  &k=100

Production Boundaries

The Java implementation should keep these as interfaces even if the first adapters are local or in-memory.

Input

Watch events

Schema validation, auth, event id dedupe, low-quality playback suppression, and late-event handling.

State

Rolling counters

Ring-buffer counters keyed by scope, metric, window, and video with event-time bucket assignment.

Output

Ranking snapshots

Durable snapshots with score version, asOf, watermark, rank quality, and deterministic ordering.

YouTube Top-K Architecture Component 3: trust boundary for ranking signals
Component 3

Validation, Deduplication, Fraud Filter

Component 3 converts untrusted raw playback activity into trusted ranking signals. It decides whether an event can affect hot top-k rankings, whether it should be rejected, or whether it should be quarantined for later review.

InputRawWatchEvent from the event gateway.
DecisionAccept, reject, quarantine, or route to backfill.
OutputValidViewEvent into Component 4.
DefaultIf trust is uncertain, do not affect hot rankings.

1. High-Level System Interactions

Component 3 sits between the gateway and the durable event stream. It uses short-lived state stores and metadata services, then emits separate streams for valid, rejected, suspicious, and late events.

3. State And Hot-Node Strategy

Event-id dedupe distributes well because event IDs are random. Hot-node risk comes from low-cardinality or viral keys such as IP, video ID, country, and creator. Component 3 avoids direct per-video counter writes and salts hot abuse counters.

4. Failure Behavior

The failure policy is conservative: if we cannot trust the event, it should not affect hot rankings. Some failures degrade to cached/simple rules; others quarantine or reject.

Node Failure Hot Ranking Behavior Follow-Up
Schema validator Bad schema or parser bug Reject malformed events; canary rollback on rate shift Dead-letter queue and schema version audit
Auth/session validator Session store unavailable Use short local cache; otherwise quarantine Replay quarantined events if trust can be proven
Deduplicator Redis shard hot or unavailable Use local dedupe fallback; uncertain events quarantine Scale shards, shorten TTL, or enable probabilistic filter
Fraud filter Rules/model unavailable Fall back to simple rules; quarantine high-risk traffic Async review and correction snapshots
Policy gate Policy metadata unavailable Use cached policy; unknown videos do not rank Refresh metadata and replay if needed
Valid-event producer Component 4 unavailable Backpressure intake; bounded buffering only Do not silently drop accepted valid events

5. Capacity Model For Component 3

These are practical planning numbers for a budget-conscious global design. They should be validated with load tests once implementation begins.

MVP Production

25k peak events/sec

4-6 Java workers, one regional cache cluster, simple rules, local dedupe, and 30-minute event-id TTL.

Serious V1

100k peak events/sec

15-25 workers globally, regional Redis/Valkey, local dedupe first, abuse counter salting, and separate audit streams.

Large Scale

1M peak events/sec

130-180 workers globally, 50-100 cache shards depending on TTL, aggressive hot-key detection, and no per-video writes here.
